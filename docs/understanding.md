# Понимание проекта — «Release любой ценой» (визуальный слой)

> Этот документ — зафиксированный **разбор** игры и её визуала перед проектированием
> структуры. Составлен по правилам (`user_input/Rules - …md`), дизайну карт
> (`user_input/Card/`) и серии уточнений с автором. Описания механик автор ещё будет
> править — здесь важна **суть и интерактивные следствия**, а не точная буква правил.

## 1. Что это за игра (кратко)

Соревновательная «take-that» карточная игра про айтишную культуру. 2–6 игроков, 15–45 мин.
**Цель:** первым собрать в своей зоне релиза три разные Release-карты (Frontend, Backend,
Database), отбив атаки, — либо остаться последним игроком.

Две колоды: **основная** (зелёные рубашки, 104 карты) и **события / AI** (фиолетовые,
21 карта; по сути — пул AI-эффектов, не истощается, тянется только при триггере AI).

**Наш scope** напоминанием: мы делаем только **визуальный слой** (интерактив, анимации,
верстка). Правила и P2P — зона друга-разработчика. Всё игровое состояние у нас — на моках,
ровно настолько, чтобы прогонять анимации/интерактив глазами. См. `CLAUDE.md`.

## 2. Дизайн карт (по факту артов)

Все карты **368×515** (стандартная карточная пропорция ~0.71). Арт — законченные PNG.

**Анатомия лица карты** (единая):
- левый-верхний угол: тег категории + иконка, цвет = категория;
- правый-верхний угол: статус-иконка (напр. **молния** у атак);
- центр: крупный заголовок + 3D-арт со свечением;
- низ: описание; иногда отдельный блок «**sudo X:**» с усиленным эффектом;
- тёмный техно-фон с сеткой/свечением, подкрашенный под категорию.

**Категории основной колоды:** Attack, Defense, Protection, Release, Operation (Git),
Support, Trigger. Отдельно — колода **AI** (12 уникальных эффектов). Цветовое кодирование
уже встроено в арт: Release — cyan, Attack — красный (+молния), Operation — оранжевый,
Protection — фиолетовый, Trigger/Error 503 — красный-danger, AI/Events — фиолетовый.

**Рубашки** — почти чёрный фон с еле заметной сеткой («экран монитора»), вертикальная
неоновая полоса (зелёная base / фиолетовая ai), лого «Release / любой ценой».

## 3. Эстетика и UX-референс

- **«Экран — это экран».** Никакого скевоморфизма: ни деревянного стола, ни фетра, ни 3D-
  перспективы. Плоские карты-панели на тёмной «экранной» поверхности. Тёмная тема, лёгкая
  сетка-грид, неоновое свечение по категориям, моноширинный шрифт (JetBrains Mono).
- **UX-референс — Hearthstone**, но только в местах, где он ложится по механике. Берём:
  1. рука веером внизу, плавно переукладывается при приходе/уходе карт;
  2. **hover → карта всплывает/увеличивается/читается** (арты детальные, в мелком виде
     нечитаемы — ховер решает именно чтение);
  3. **drag-to-play** в зону/на цель;
  4. **targeting-стрелка** к цели (у игры много адресных действий);
  5. зелёное свечение «можно сыграть» (временный цвет — брендовый зелёный);
  6. полёты карт: добор / розыгрыш / сброс с «весом» и импактом;
  7. (условно) лёгкий parallax-наклон карты под курсором — только если в вебе не выйдет
     кринжово; ловим грань на тонкой настройке.
- **Не берём из Hearthstone** то, что не ложится: мана-кривую, бой существ на доске, и т.п.
  Самый интерактивный момент игры — окно реакции на релиз — у HS аналога не имеет.

## 4. Рендер карты — открытое архитектурное решение

- Сейчас: карта = **готовый PNG** (лицо + рубашка) — быстрый путь.
- Хотим **сохранить возможность подмены** на карту, **собранную из кода** (больше
  возможностей: динамика, анимация внутренностей). Карту трактуем за абстракцией, чтобы
  «лицо» можно было заменить, не переписывая окружение.
- **Критерий выбора** — стабильность и производительность у пользователей (в т.ч. на слабых
  устройствах). Решаем по замерам, не на вкус. Решение **отложено**.

## 5. Состояния-оверлеи (рисуем мы, поверх арта)

Арт статичен. Всё «живое» — наша зона: подсветка «можно сыграть», выбор/hover, «атакована»,
«защищена (Code Review)», «заморожена» (DDoS, 1 раунд), «нейтрализована», перелёт в сброс,
раздача, флип рубашка↔лицо. Временный цвет подсветки — **брендовый зелёный**, пока не дойдём
до эффектов отдельно.

## 6. Карта интерактивных моментов

1. **Ход:** играть карты → обязательный добор 1 → играть карты → «PUSH» (передача по часовой).
2. **Релиз** (сброс 1 карты с руки) → открывает окно реакции (если без Code Review).
3. **Окно реакции** (см. §7) — ключевой и уникальный момент.
4. **Комбо-розыгрыш** (см. §8): Sudo + карта, Code Review + Release.
5. **Проактивные атаки в свой ход** по руке оппонента (укради/запроси карту) — прицел стрелкой.
6. **Триггеры при доборе:** AI (показать всем → тянуть AI-эффект → разыграть),
   Error 503 (показать → нейтрализовать или выбывание).
7. **Адресные утилиты:** Security Bug (запрос конкретной карты), Git Cherry-pick (из сброса),
   System Upgrade (все скидывают по 1), DDoS (снести Monitoring / вернуть релиз в руку).
8. **Манипуляции колодами:** Git Branch (1→2 стопки + вариативность добора, см. ниже),
   Git Merge (слить), Git Rebase (подсмотреть/переставить топ-3 приватно), сброс→колода.
9. **Состояния зоны релиза:** свежий / защищён (Code Review) / заморожен (DDoS) / уничтожен.
10. **Выбывание игрока** (нейтрализованный Error 503).
11. **Пре-гейм:** хост выбирает режимы; в игре — возможность **посмотреть активные режимы**.
12. **Окно истории действий** — хронология разыгранных карт, открывается **сбоку как «чат»**.
13. **Обёртка:** лоадер → лобби → стол.

**Вариативность разделённой колоды (Git Branch):** при разделённой колоде добора механика
будет вариативной — либо игрок **выбирает, из какой стопки** тянуть, либо берутся **верхние
карты со всех стопок**. Точную интерпретацию автор ещё дорабатывает.

## 7. Окно реакции на релиз (модель)

- Открывается при выкладывании Release **без** Code Review. Таймер первой общей реакции —
  **15 сек**. Соперники могут вбросить атаку либо **спасовать**.
- **Приём или отбой — по выбору игрока.** Даже имея карту защиты, можно «проглотить» атаку
  (человеческий фактор). Карты, которыми **можно отбить**, мы **подсвечиваем**.
- **«Пас» ≠ отказ от атаки.** Пас = «не против, чтобы окно закрылось раньше 15 сек». Окно
  закрывается, когда таймер истёк **или** спасовали все. Спасовавший может **передумать** и
  атаковать, пока окно открыто.
- **Многораундовость:** после того как атаку **отбили**, открывается **новое окно 10 сек**
  «продолжить атаку на этот релиз». Пас в 1-м раунде **не лишает** атаки во 2-м (окна
  независимы).
- **Без стека и приоритетов:** одновременные атаки — нерелевантный сценарий. Всё строго
  **последовательно**: атака → принял/отбил → новое окно. Это упрощает и логику, и оверлей.
- **Видимость:** момент **выделяется для всех** — релиз в центре как «горячая точка», таймер.
- **Открытый визуальный вопрос:** куда складывать карты реакции — общая куча у релиза vs
  сразу в сброс отбитых. Решим на этапе дизайна.

## 8. Комбо-розыгрыш: Sudo и Code Review

Оба — «**две карты как одно действие**», заявляются **в момент розыгрыша** (не прилепляются к
давно лежащему объекту, поэтому «открепление», как в Hearthstone Magnetic, не нужно).

- **Жест (для обоих):** карта разыгрывается из руки → **стрелкой выбираешь карту-партнёра** (с
  возможностью отмены/передумать) → подтверждение сопровождается **снап-фидбэком** (карты
  эффектно «сцепляются») + появляется sudo-свечение/бейдж. (Выбор — из варианта-1, снап —
  визуальный приём из Magnetic; необратимость Magnetic не тащим.)
- **Sudo** целится в **другую карту руки** с sudo-эффектом, разблокируя её усиленный вариант.
- **Code Review** соединяется с **Release в момент его выкладывания**. Релиз с Code Review
  **нельзя атаковать вовсе** → для такого релиза **окно реакции не открывается**. На столе
  карта Code Review лежит **под** релизом, обозначая защиту (визуализация не финальная).

## 9. Открытые / отложенные вопросы

- **Стек/сборка проекта** — сознательно отложено (чтобы не связать руки коллеге по логике).
- **Рендер карты** image vs code-composed — решаем по стабильности/перформансу (§4).
- **Финальная визуализация Code Review** на столе (сейчас «лежит под релизом» — рабочее).
- **Куча карт реакции vs мгновенный сброс** — визуальный вопрос (§7).
- **Интерпретация разделённой колоды** (Git Branch) — автор дорабатывает (§6).
- **Эффекты/свечения** — отдельный разговор на этапе дизайна (пока брендовый зелёный).
